## 2. 上部カテゴリタブ

### 現状の問題

`LINE_CATEGORY_TABS` が旧取引チャット用（個人 / 求人 / ワーカー / スキル / 商品 …）のまま。Phase 1 で取引相手スレッドは一覧から除外済みのため、**求人・ワーカー等を選んでもほぼ空**になり、ユーザー体験と Phase 3 の 3 ルーム構造が一致しない。

### 案 A — `すべて / プラット / 安否 / 運営 / 友達`

| タブ | 表示対象 |
|------|----------|
| すべて | 6 件すべて |
| プラット | `TASFULプラット通知`（`official_platform`） |
| 安否 | `TASFUL安否通知`（`official_anpi`） |
| 運営 | `TASFUL運営通知`（`official_tasful`） |
| 友達 | `友達` ハブ + 友達スレッド（将来） |

**すべてのみ:** TASFUL AI、TASFULサポート（静的ハブ。通知ルームではない）

### 案 B — `すべて / 通知 / 友達`

| タブ | 表示対象 |
|------|----------|
| すべて | 6 件すべて |
| 通知 | 3 公式通知ルームのみ |
| 友達 | `友達` ハブのみ |

---

## 3. 推奨: **案 A**

### 理由

1. **Phase 3 の 3 ルームと 1:1** — プラット / 安否 / 運営はすでに一覧上の独立行。タブ名とルーム名が対応し、初見でも理解しやすい。
2. **左ナビとの役割分担が明確** — 左の「プラット」「安否」= ショートカット、上のカテゴリ = 一覧内フィルタ。案 B の「通知」タブは左ナビの新項目と意味が重なり、「通知を廃止する」方針とも矛盾しやすい。
3. **案 B の「通知」は粒度が粗い** — 3 ルームをまとめるだけなら「すべて」で足り、運営だけ見たい需要を満たせない。
4. **プレビュー行の【求人】【安否】プレフィックス**（Phase 3）と、案 A のタブが同じ語彙で揃う。

### 案 B が向くケース

- カテゴリ UI を極限まで減らしたい（3 タブ固定）
- ユーザーに「通知系 vs 友達」だけ意識させれば十分

通知センターとして Phase 3 まで進めている以上、**案 A を推奨**。

### 案 A 採用時の実装メモ

- フィルタは旧 `chatChannels`（job / worker …）ではなく **`roomId` または `_notificationCenterKind`** で判定（`official_platform` の `_talkChannel` は `system` のため channel ベースでは区別不可）。
- ⚙ 詳細フィルターは通知センター向けに縮小（期間・未読・キーワード中心。取引カテゴリチップは通知タブ側のみ残すか別途整理）。
